# NGT Metadata Reference — `AiTestingDefinition` XML

The deployed form of every NGT (Agentforce Studio) test suite is an `AiTestingDefinition`
metadata file. It is a first-class artifact you can **retrieve, hand-edit, source-control, and
redeploy** directly. This reference documents that metadata so you can work at the XML layer —
the only path for features the higher-level `sf agent test create` spec flow doesn't expose
(custom scorers, the org-gated `hallucination_score` scorer, and any field not emitted by the
spec compiler).

> **NGT vs. legacy Testing Center.** This document covers **NGT** (Agentforce Studio), whose
> metadata type is `AiTestingDefinition` and whose runner is `agentforce-studio`. It is distinct
> from the legacy **Testing Center** path documented in `references/batch-testing.md`, which uses
> `AiEvaluationDefinition` and the default runner. The two are separate metadata types with
> separate schemas — don't mix their fields.

> **Everything here is CLI-driven.** Retrieve and deploy `AiTestingDefinition` with
> `sf project retrieve start` / `sf project deploy start`; create and run suites with
> `sf agent test`. The NGT Connect API endpoints (`POST /einstein/ai-testing/runs`,
> `GET .../runs/:runId`, `GET .../runs/:runId/results`) are what these CLI commands call
> **internally** — this skill never issues raw API calls against them. If you find yourself
> reaching for `curl` against `/einstein/ai-testing/...`, stop and use the corresponding
> `sf agent test` subcommand instead. The CLI owns auth, polling, and result formatting.

## NGT support gate

NGT (`AiTestingDefinition` + the `agentforce-studio` runner) requires:

- **`sf` CLI** with the `@salesforce/agents` library new enough to expose
  `--test-runner agentforce-studio` on `sf agent test create`. Probe with
  `sf agent test create --help | grep -i test-runner`; if the flag is absent, the CLI is too old
  and only the legacy Testing Center path is available.
- **Project `sourceApiVersion >= 66.0`** (in `sfdx-project.json`) — the API version that
  introduced the `AiTestingDefinition` metadata type. Deploys below 66.0 fail with an
  "unknown metadata type" error.
- An org with NGT enabled. If a deploy or run is rejected on entitlement, the org isn't gated for
  NGT (or, for custom scorers and `hallucination_score`, lacks those specific gates — see below).

## When to work at the XML layer vs the spec layer

| Situation | Layer | Tool |
|---|---|---|
| Authoring a new suite from scratch | Spec (YAML) | `sf agent test create --test-runner agentforce-studio --spec spec.yaml` |
| Quick edit to an existing deployed suite | Spec round-trip | `sf agent generate test-spec --from-definition <xml>` → edit → `create --force-overwrite` |
| Suite already lives in source control as XML | XML | `sf project deploy start --metadata AiTestingDefinition:<Name>` |
| Diffing two suite versions in a PR | XML | `sf project retrieve start` then `git diff` |
| Attaching a **custom scorer** (`scorerType: Custom`) — not representable in the spec | XML | Hand-edit, then `sf project deploy start` |
| Any other field the spec compiler doesn't emit | XML | Hand-edit, then `sf project deploy start` |
| CI that deploys metadata bundles wholesale | XML | `sf project deploy start` over the whole `aiTestingDefinitions/` dir |

The spec (YAML) path is the default for authoring. Reach for the XML when the suite is **already**
metadata — retrieved from an org, committed to a repo, or generated by another tool — or when you
need a field the spec compiler doesn't emit (custom scorers being the headline case).

## File layout

```text
force-app/main/default/aiTestingDefinitions/
  <TestSuiteName>.aiTestingDefinition-meta.xml
```

- Folder: `aiTestingDefinitions/` (note the plural, lowercase-first).
- Suffix: `.aiTestingDefinition-meta.xml`.
- The file's base name (`<TestSuiteName>`) is the metadata **API name / DeveloperName**. It must
  match the root `<name>` element inside the file (see the `name` rule below) — Salesforce keys on
  both and a mismatch fails deploy.
- Metadata API type name: `AiTestingDefinition` (use this verbatim in `--metadata` flags).
- Requires project `sourceApiVersion >= 66.0` (see **NGT support gate** above).

## Root element: `AiTestingDefinition`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<AiTestingDefinition xmlns="http://soap.sforce.com/2006/04/metadata">
    <description>My Sample Tests</description>
    <name>TestAiTestingDefinition</name>
    <subjectName>Agentforce_for_Salesforce</subjectName>
    <subjectType>AGENT</subjectType>
    <subjectVersion>v1</subjectVersion>
    <testCase>
        <!-- ... one or more, see below ... -->
    </testCase>
</AiTestingDefinition>
```

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `description` | string | No | Free-form. Compiles to `MasterLabel` — **max 100 characters**. Over-length fails deploy with `Label: data value too large: ... (max length=100)`. |
| `name` | string | Yes | The suite **DeveloperName**. Alphanumeric + underscore, starts with a letter, no spaces, no double/trailing underscore. Must equal the file's base name. Invalid chars fail with `The AI Test Suite Definition API Name can only contain underscores and alphanumeric characters...`. |
| `subjectName` | string | Yes | API name (`BotDefinition` DeveloperName) of the agent under test. |
| `subjectType` | enum | Yes | `AGENT`. |
| `subjectVersion` | string | No | e.g. `v1`, `LATEST`. Defaults to `v1`. Pin only when testing a specific saved version. |
| `testCase` | `AiTestCase[]` | Yes | One `<testCase>` element per scenario. At least one required. |

> **`<name>` is the field that maps to the spec `name:` / `--api-name` triad.** When you author
> via the spec, `sf agent test create --api-name X` must match the spec `name: X`, which becomes
> `<name>X</name>` here. When you deploy XML directly, the file name, `<name>`, and any
> `--metadata AiTestingDefinition:X` reference must all agree on `X`.

## `AiTestCase` (`<testCase>`)

```xml
<testCase>
    <number>1</number>
    <inputs>
        <utterance>Summarize the Global Media account</utterance>
    </inputs>
    <scorer>
        <name>topic_sequence_match</name>
        <expectedValue>OOTBSingleRecordSummary</expectedValue>
    </scorer>
    <scorer>
        <name>action_sequence_match</name>
        <expectedValue>['IdentifyRecordByName']</expectedValue>
    </scorer>
    <scorer>
        <name>bot_response_rating</name>
        <expectedValue>Summarization of the Global Media account</expectedValue>
    </scorer>
    <scorer>
        <name>conciseness</name>
    </scorer>
</testCase>
```

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `number` | int | Yes | 1-based sequence index. **Globally unique and contiguous** across the file. Multi-input cases (see below) fan out into multiple `<testCase>` elements that each consume a `number`. |
| `inputs` | `AiTestCaseInput` | Yes | The utterance plus optional context/history. See below. |
| `scorer` | `AiTestCaseScorer[]` | Yes | One `<scorer>` per assertion. At least one required. |

> **Note on "multi-input" cases.** A spec may let one scorer block apply to several utterances (a
> paraphrase group). There is **no** multi-utterance container at the XML layer — the compiler
> fans each utterance out into its own `<testCase>` with a copy of the scorer set and its own
> `<number>`. When hand-editing XML, replicate the `<testCase>` block per utterance.

## `AiTestCaseInput` (`<inputs>`)

```xml
<inputs>
    <utterance>Show me my open cases</utterance>
    <contextVariable>
        <variableName>UserId</variableName>
        <variableValue>005000000000123</variableValue>
    </contextVariable>
    <contextVariable>
        <variableName>TenantRegion</variableName>
        <variableValue>us-east-1</variableValue>
    </contextVariable>
    <conversationHistory>
        <role>user</role>
        <message>I need to check my mortgage status</message>
        <index>0</index>
    </conversationHistory>
    <conversationHistory>
        <role>agent</role>
        <topic>identity_verification</topic>
        <message>Sure — what is your email address on file?</message>
        <index>1</index>
    </conversationHistory>
</inputs>
```

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `utterance` | string | Yes | The user input message the agent responds to. |
| `contextVariable` | `AiTestCaseContextVar[]` | No | Session context injected before the utterance. Repeat the element per variable. |
| `conversationHistory` | `AiTestCaseConvHistory[]` | No | Prior turns that set up a multi-turn test. Repeat the element per turn. Required when any scorer is `task_resolution`. |

### `AiTestCaseContextVar` (`<contextVariable>`)

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `variableName` | string | Yes | Context variable API name as the agent expects it. |
| `variableValue` | string | Yes | Value injected into session context. Use synthetic/test data — never real PII. |

### `AiTestCaseConvHistory` (`<conversationHistory>`)

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `role` | string | Yes | `user` or `agent` only. |
| `message` | string | Yes | The turn's text. |
| `topic` | string | Conditional | **Required on `agent` turns** — the subagent/topic that produced that turn. Omit on `user` turns. |
| `index` | int | No | Zero-based turn order. **All-or-nothing per case**: either every turn carries an `index` or none do. |

## `AiTestCaseScorer` (`<scorer>`)

A `<scorer>` is either an **OOTB scorer** (a built-in name from the catalog below) or a **custom
scorer** (a customer-defined evaluator referenced by its DeveloperName with
`<scorerType>Custom</scorerType>`).

```xml
<!-- OOTB scorer: name is a catalog key; expectedValue per the catalog -->
<scorer>
    <name>topic_sequence_match</name>
    <expectedValue>OOTBSingleRecordSummary</expectedValue>
</scorer>

<!-- Custom scorer: name is the custom evaluator's DeveloperName; scorerType is Custom -->
<scorer>
    <name>CustomScorer2</name>
    <scorerType>Custom</scorerType>
</scorer>
```

| Element | Type | Required | Notes |
|---|---|:-:|---|
| `name` | string | Yes | For OOTB scorers, a key from the OOTB scorer catalog below. For custom scorers, the **DeveloperName of the custom evaluator** (`AiAgentScorerDefinition`) registered in the org. |
| `scorerType` | enum | Conditional | `Standard` (OOTB) or `Custom`. **Omit for OOTB scorers** — they default to `Standard`. **Required (`Custom`) for custom scorers** — without it the runtime resolves `name` against the OOTB catalog and fails on a name that isn't there. |
| `expectedValue` | string | Conditional | OOTB: required for scorers the catalog marks "needs expected" (the deterministic matchers and `bot_response_rating` / `response_match`); omit for quality/numeric scorers. Custom: **omit** — the custom evaluator owns its own pass/fail logic; there is no skill-supplied expected value. |

### OOTB scorer catalog

The shipped catalog has 11 OOTB scorers. The authoritative list lives in the installed
`@salesforce/agents` library (`lib/ngtScorerCatalog`) and may grow — when the CLI is available,
prefer reading it at runtime over trusting this table. As of writing:

| `name` | Grade class | Needs `expectedValue`? | `expectedValue` format |
|---|---|:-:|---|
| `topic_sequence_match` | PASS_FAIL (deterministic) | Yes | GenAiPlugin/subagent DeveloperName (e.g. `order_status`) |
| `action_sequence_match` | PASS_FAIL (deterministic) | Yes | Python-list string of **Level 2 invocation** names (e.g. `['lookup_order']`); `[]` asserts no actions. Order-sensitive |
| `agent_handoff_match` | PASS_FAIL (deterministic) | Yes | Target Bot's `BotDefinition` DeveloperName |
| `bot_response_rating` | LLM_PASS_FAIL | Yes | Free-text rubric/exemplar the LLM judges against |
| `response_match` | LLM_PASS_FAIL | Yes | Free-text rubric/exemplar the LLM judges against |
| `coherence` | LLM_0_100 | No | — |
| `conciseness` | LLM_0_100 | No | — |
| `factuality` | LLM_0_100 | No | — |
| `completeness` | LLM_0_100 | No | — |
| `task_resolution` | LLM_0_5 | No | Requires `<conversationHistory>` on the case |
| `output_latency_milliseconds` | NUMERIC | No | — |

> **`hallucination_score` — metadata enum vs. shipped catalog.** The `AiTestingDefinition`
> metadata schema lists `hallucination_score` as a valid `<scorer><name>` value, but it is **not**
> in the 11-row OOTB catalog shipped by `@salesforce/agents` (which carries `factuality` as the
> closest hallucination-detection scorer). Treat `hallucination_score` as org-gated: it deploys
> only where the org has enabled it, and the spec compiler won't scaffold it. If you need it,
> hand-edit the XML and confirm against the runtime catalog before deploying. When in doubt, use
> `factuality`.

## Custom scorers (`scorerType: Custom`) — XML-only

Custom scorers let you evaluate a test case against an evaluator you define in the org, rather than
one of the 11 OOTB scorers. They are **not** exposed in the NGT spec compiler — the XML metadata
layer is the only way to attach them to a test suite today. This is the single biggest reason to
work at the XML layer.

There are **two separate metadata types** involved, and it's important not to conflate them:

1. **`AiAgentScorerDefinition`** — *defines* the custom scorer (its engine, scoring scale, pass/fail
   outcomes, agent association). This is the heavyweight metadata; see
   **Defining the custom scorer** below.
2. **`AiTestingDefinition`** — *references* the custom scorer from a test case. The reference is
   **just the scorer's DeveloperName plus `<scorerType>Custom</scorerType>`** — nothing else. The
   test suite does not redeclare the engine, scale, or outcomes; all of that lives in the
   `AiAgentScorerDefinition`.

This section covers the reference (the `<scorer>` block) first, then the definition.

### Referencing a custom scorer from `AiTestingDefinition`

A custom scorer reference is a `<scorer>` block with exactly two child elements — no `<expectedValue>`:

```xml
<scorer>
    <name>CustomScorer2</name>
    <scorerType>Custom</scorerType>
</scorer>
```

| Element | Required | Value |
|---|:-:|---|
| `name` | Yes | The **DeveloperName of the `AiAgentScorerDefinition`** (e.g. `CustomScorer2`). This is *not* an OOTB catalog key — it's the API name of the custom scorer metadata deployed to the org. |
| `scorerType` | Yes | Literal `Custom`. This is the flag that tells the runtime to resolve `name` against the org's custom-scorer metadata (`AiAgentScorerDefinition`) instead of the OOTB catalog. Drop it and the run fails resolving `CustomScorer2` against OOTB names. |
| `expectedValue` | No | **Do not set.** The `AiAgentScorerDefinition` encapsulates the engine and pass/fail outcomes; there is no test-author-supplied expected value. Adding one is ignored at best and a validation error at worst. |

`scorerType` is an enum with two values:

| `scorerType` | Meaning | When to write it |
|---|---|---|
| `Standard` | Built-in OOTB scorer (the 11-row catalog) | **Never write it explicitly** — OOTB scorers omit `scorerType` and default to `Standard`. Writing it is harmless but noise. |
| `Custom` | Customer-defined evaluator resolved by DeveloperName | **Always write it** on a custom scorer. |

### Authoring workflow

Custom scorers are XML-only and org-gated, so the flow is always confirm-exists/retrieve/hand-edit/deploy.
The custom scorer is **metadata** (`AiAgentScorerDefinition`), not a queryable SObject — list it with
`sf org list metadata`, not SOQL:

```bash
# 1. Confirm the custom scorer exists in the org under the DeveloperName you'll reference.
#    Custom scorers are org-resident metadata of type AiAgentScorerDefinition.
sf org list metadata --metadata-type AiAgentScorerDefinition -o <org> --json \
  | python3 -c "import json,sys; [print(m['fullName']) for m in (json.load(sys.stdin).get('result') or [])]"
#    The DeveloperName you put in <name> must match one of these fullNames.

# 2. Retrieve the suite XML (or author it fresh), then hand-edit to add the custom <scorer> block.
sf project retrieve start --json --metadata "AiTestingDefinition:<Name>" -o <org>

# 3. Deploy the edited test suite XML.
sf project deploy start --json --metadata "AiTestingDefinition:<Name>" -o <org>

# 4. Run via the same CLI as any other suite.
sf agent test run --json --api-name <Name> --wait 10 --result-format json -o <org>
```

If the scorer doesn't exist yet, define and deploy the `AiAgentScorerDefinition` first (see
**Defining the custom scorer** below), then reference it from the test suite.

A custom scorer can sit alongside OOTB scorers in the same `<testCase>` — mix freely:

```xml
<testCase>
    <number>1</number>
    <inputs>
        <utterance>Summarize the Global Media account</utterance>
    </inputs>
    <scorer>
        <name>topic_sequence_match</name>
        <expectedValue>OOTBSingleRecordSummary</expectedValue>
    </scorer>
    <scorer>
        <name>CustomScorer2</name>
        <scorerType>Custom</scorerType>
    </scorer>
</testCase>
```

### Validation and failure modes

- **Unregistered custom scorer** — if `<name>` doesn't resolve to a deployed `AiAgentScorerDefinition`
  in the org, the test-suite deploy/run fails at the org-side resolution gate (the same place a
  misspelled OOTB name fails). Local spec validation does not catch this — failure surfaces at
  `sf project deploy start` / `sf agent test create` / `sf agent test run`. Deploy the
  `AiAgentScorerDefinition` **before** the test suite that references it.
- **Missing `<scorerType>Custom</scorerType>`** — the runtime treats `name` as an OOTB key, doesn't
  find it in the 11-row catalog, and fails resolving it. Symptom looks like a "misspelled OOTB
  scorer" error even though the name is a perfectly valid custom scorer.
- **`<expectedValue>` on a custom scorer** — the custom evaluator owns its grading; an
  author-supplied expected value has no effect and may be rejected. Leave it off.
- **Results shape** — a custom scorer reports through the same `testScorerResults[]` /
  `scorerResponse` envelope as OOTB scorers (`scorerName` = your DeveloperName; `status` =
  PASS/FAIL; `score`/`reasoning` if the evaluator emits them). See
  `references/test-report-format.md` for the result-shape reference.

### Round-tripping

`sf agent generate test-spec --from-definition <xml>` round-trips OOTB scorers back to a spec, but
the spec schema has no representation for custom scorers — so a custom `<scorer>` block is
**dropped or flagged** on the way back. Treat the XML as the source of truth for any suite that
uses a custom scorer, and keep it in source control directly rather than regenerating it.

### Defining the custom scorer (`AiAgentScorerDefinition`)

The custom scorer itself is a separate metadata type, `AiAgentScorerDefinition`. The
`AiTestingDefinition` only ever references it by DeveloperName (above) — you author and deploy the
scorer definition once, then reference it from any number of test suites.

**File layout**

```text
force-app/main/default/aiAgentScorerDefinitions/
  <ScorerDeveloperName>.aiAgentScorerDefinition-meta.xml
```

- Folder: `aiAgentScorerDefinitions/` (plural).
- Metadata API type: `AiAgentScorerDefinition`.
- The file base name is the **DeveloperName** — this is the exact string you put in the test
  suite's `<scorer><name>`. (In `package.xml` it's the `<members>` value.)
- Requires Metadata API `66.0+`.

**Schema** — the definition carries one or more `<scorerVersion>` blocks; each version pins an engine
(prompt template or formula expression), its output outcomes, and an optional numeric scale:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<AiAgentScorerDefinition xmlns="http://soap.sforce.com/2006/04/metadata">
    <inputScope>Session</inputScope>          <!-- Session | Interaction | Moment -->
    <dataType>Number</dataType>               <!-- Text | Number -->
    <scorerVersion>
        <versionNumber>1</versionNumber>
        <status>Available</status>            <!-- Draft | Available | Archived -->
        <description>Checks the response stays on-policy</description>
        <label>On-policy check</label>
        <agentAssociation>
            <isActive>true</isActive>
            <agentApiName>Copilot_for_Salesforce</agentApiName>
        </agentAssociation>
        <engine>
            <engineType>PromptTemplate</engineType>   <!-- PromptTemplate | Expression -->
            <engineRef>My_Scorer_Prompt_Template</engineRef>  <!-- API name of the prompt template -->
            <!-- For engineType Expression, use <engineValue> with a formula instead of <engineRef> -->
        </engine>
        <outputEnumValue>
            <isFallback>true</isFallback>
            <outcomeType>Fail</outcomeType>   <!-- Fail | Pass | NotApplicable -->
            <value>0</value>
        </outputEnumValue>
        <outputEnumValue>
            <isFallback>false</isFallback>
            <outcomeType>Pass</outcomeType>
            <value>1</value>
        </outputEnumValue>
        <specification>
            <valueSpecification>
                <min>0</min>
                <max>5</max>
                <step>1</step>
                <threshold>3</threshold>
            </valueSpecification>
        </specification>
    </scorerVersion>
</AiAgentScorerDefinition>
```

| Element | Notes |
|---|---|
| `inputScope` | `Session`, `Interaction`, or `Moment` — what slice of the conversation the scorer sees. |
| `dataType` | `Text` or `Number` — the type the scorer emits. |
| `scorerVersion` | One or more. Versions are **append-only** (see deploy rules below). |
| `scorerVersion.status` | `Draft`, `Available`, or `Archived`. Only `Available` versions are usable by a test run. |
| `agentAssociation` | Binds the scorer to an agent (`agentApiName`) and toggles it (`isActive`). |
| `engine.engineType` | `PromptTemplate` (LLM-judged, references a prompt template via `engineRef`) or `Expression` (formula via `engineValue`). |
| `outputEnumValue` | Maps each emitted `value` to an `outcomeType` (`Pass`/`Fail`/`NotApplicable`); exactly one should be `isFallback: true`. |
| `specification.valueSpecification` | For numeric scorers — `min`/`max`/`step` define the scale, `threshold` the pass cutoff. |

**Entitlement / gating.** Custom scorers are gated. The org needs one of the gater flags
(`Gater.com.salesforce.agentforce.agentforceEvals` or
`Gater.com.salesforce.agentforce.agentforceCustomEvalsTestingCenter`) and one of the org perms
(`GenerativeAIAgent.orgHasAnyCopilot` or `OrgPreferences.AgentforceSTDMEnabled`). If a deploy is
rejected on entitlement, this gating is why.

**Deploy/update rules.** Once deployed you can **add new `<scorerVersion>` blocks but not delete
existing ones**; you may update a version's `status` and an `agentAssociation`'s `isActive`. Deploy
and retrieve with the standard CLI:

```bash
sf project deploy start --json --metadata "AiAgentScorerDefinition:<ScorerDeveloperName>" -o <org>
sf project retrieve start --json --metadata "AiAgentScorerDefinition:<ScorerDeveloperName>" -o <org>
```

> Deploy the `AiAgentScorerDefinition` **and** make a `<scorerVersion>` `Available` before deploying
> or running an `AiTestingDefinition` that references it — a reference to a non-existent or
> Draft-only scorer fails at run time. `Available` is currently the only run-usable status.

## CLI workflow for the metadata path

All steps below use `sf` — no raw API calls.

```bash
# 1. Retrieve an existing suite from an org into the project
sf project retrieve start --json \
  --metadata "AiTestingDefinition:TestAiTestingDefinition" \
  -o <org>
# Lands at force-app/main/default/aiTestingDefinitions/TestAiTestingDefinition.aiTestingDefinition-meta.xml

# 2. (Optional) Round-trip the XML to a spec to edit in the compact form, then back
sf agent generate test-spec \
  --from-definition force-app/main/default/aiTestingDefinitions/TestAiTestingDefinition.aiTestingDefinition-meta.xml
#   -> edit the spec, then re-create with --force-overwrite (custom scorers are NOT preserved on this round-trip)

# 3. Or hand-edit the XML directly, then deploy it
sf project deploy start --json \
  --metadata "AiTestingDefinition:TestAiTestingDefinition" \
  -o <org>

# 4. Run it — runner auto-detected from the suite name (pass --test-runner agentforce-studio
#    only if the name is ambiguous). The CLI calls POST /einstein/ai-testing/runs internally.
sf agent test run --json --api-name TestAiTestingDefinition --wait 10 --result-format json -o <org>

# 5. Read results via the CLI's built-in formatters (it calls GET .../runs/:runId/results internally)
sf agent test results --json --job-id <runId> --result-format human -o <org>
```

For the result-shape details (`testScorerResults[]`, the JSON-encoded `scorerResponse` /
`subjectResponse`), see `references/test-report-format.md`.

## Pre-deploy checklist for hand-edited XML

Before `sf project deploy start` on an XML you edited by hand:

- [ ] `<name>` equals the file's base name and the `--metadata` reference.
- [ ] `<name>` is a valid DeveloperName (no spaces, alphanumeric + underscore, starts with letter).
- [ ] `<description>` ≤ 100 characters.
- [ ] `<number>` values are contiguous and unique starting at 1.
- [ ] Every `<testCase>` has ≥1 `<inputs>` and ≥1 `<scorer>`.
- [ ] Every `agent` turn in `<conversationHistory>` has a `<topic>`.
- [ ] `<index>` is present on all `<conversationHistory>` turns of a case, or none.
- [ ] Any case with a `task_resolution` scorer has `<conversationHistory>`.
- [ ] OOTB scorer `<name>` values are in the catalog (read the runtime catalog when the CLI is
      available); `<expectedValue>` present wherever the catalog says it's needed.
- [ ] Custom scorers have `<scorerType>Custom</scorerType>`, a `<name>` that resolves to a
      registered org-resident custom scorer, and **no** `<expectedValue>`.
- [ ] Project `sourceApiVersion >= 66.0`.
